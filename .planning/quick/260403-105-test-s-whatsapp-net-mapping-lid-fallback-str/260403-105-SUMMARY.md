## Implementation Details

### Cache Population Flow (Verified by Tests)

**Webhook Call Sites:**
1. Group message handler (webhook/route.ts line 229)
2. GROUP_PARTICIPANTS_UPDATE join handler (webhook/route.ts line 643)

**Flow:**
```
@s.whatsapp.net sender sends message/joins group
  → upsertGroupSenderCache(groupJid, senderJid, normalizedPhone)
  → groupSenderCache table stores mapping
  → Members API calls getGroupSendersByGroupJid(groupJid)
  → Builds cachePhoneToName map
  → Enriches participants with voter names
```

**@lid Handling:**
```
@lid participant detected
  → isPhoneJid guard skips cache (NOT a real phone)
  → Members API calls getLidMapping(groupJid, lidJid)
  → Returns operator-curated voterName if mapped
  → Otherwise returns null (WhatsApp privacy limitation)
```

### Manual Mapping Usage

**When to Use:**
Operators manually create entries when they identify a persistent @lid participant:
1. Check group members list — see @lid entries with null voterName
2. Identify the person via other context (message content, known contacts)
3. Create manual mapping via direct DB insert or future admin UI
4. Members API now shows mapped name for that @lid

**Example:**
```typescript
await upsertLidManualMapping(
  '120363123456@g.us',           // Group JID
  '184052367761544@lid',         // @lid JID
  'João Silva',                  // Voter name
  'uuid-of-joao-silva-voter',    // Optional: link to voter record
  'Identified by profile photo', // Optional: notes
  'admin@example.com'            // Optional: who created it
);
```
