   ```sql
     name TEXT PRIMARY KEY,
     locked_at TIMESTAMPTZ NOT NULL,
     expires_at TIMESTAMPTZ NOT NULL
   );
   ```

3. Create `src/lib/cron-lock.ts` with:
   - `acquireCronLock(name: string, ttlMs: number): Promise<boolean>` — Uses an UPSERT with a WHERE condition: INSERT if no row exists, or UPDATE if existing row's `expires_at < NOW()`. Returns true if lock acquired, false if already locked by another run. Use raw SQL via `db.execute(sql\`...\`)` for atomic upsert:
     ```sql
     INSERT INTO cron_locks (name, locked_at, expires_at)
     VALUES ($name, NOW(), NOW() + $ttl * INTERVAL '1 millisecond')
     ON CONFLICT (name) DO UPDATE
       SET locked_at = NOW(), expires_at = NOW() + $ttl * INTERVAL '1 millisecond'
       WHERE cron_locks.expires_at < NOW()
     ```
     Check `rowCount > 0` to determine if lock was acquired.
   - `releaseCronLock(name: string): Promise<void>` — DELETE FROM cron_locks WHERE name = $name
   - `withCronLock<T>(name: string, ttlMs: number, fn: () => Promise<T>): Promise<{ locked: false } | { locked: true; result: T }>` — Wrapper that acquires lock, runs fn, releases lock in finally block. Returns `{ locked: false }` if lock not acquired (another run in progress). The TTL acts as a safety valve: if a cron crashes without releasing, the lock expires after ttlMs.

4. Run migration: `npx drizzle-kit push`

**B. Wrap the handler body with `withCronLock`:**

Pattern for each cron (example with send-queue):

```typescript
import { withCronLock } from '@/lib/cron-lock';

export const maxDuration = 300;

export async function GET(request: NextRequest) {
  // Auth check stays OUTSIDE the lock (fast-fail unauthorized)
  // ... existing auth code ...

  const lockResult = await withCronLock('send-queue', 360000, async () => {
    // ... move ALL business logic inside here ...
    return { /* existing response data */ };
  });

  if (!lockResult.locked) {
    return NextResponse.json({
      message: 'Execução anterior ainda em andamento',
      skipped: true,
    }, { status: 200 }); // 200, not 409 — cron schedulers retry on non-2xx
  }

  return NextResponse.json(lockResult.result);
}
```

Key rules:
- Auth check stays OUTSIDE the lock (fast-fail)
- ALL business logic goes INSIDE the withCronLock callback
- Return 200 when skipped (not 409) so cron schedulers don't retry
- The lock name matches the cron folder name (e.g., "send-queue", "ai-profile")
- For `ai-profile` which uses POST, wrap the POST handler the same way
