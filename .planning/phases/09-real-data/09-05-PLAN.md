<task type="auto">
  <name>Task 1: Real audience calculation + segment voter materialization</name>
  <files>src/app/api/segments/route.ts, src/app/segmentacao/page.tsx</files>
  <action>
  **API: Add audience preview endpoint**
  1. In `/api/segments/route.ts`, add support for a `?action=preview` query parameter on POST (or create a separate handler)
  2. The preview accepts `{ filters: object }` and returns `{ count: number, voterIds: string[] }` by querying the voters table:
     - Parse filters object (same format stored in segments.filters JSON)
     - Build a Drizzle `where` clause dynamically based on filter conditions:
       - `zone` filter: `eq(voters.zone, value)` or `inArray(voters.zone, values)`
       - `section` filter: `eq(voters.section, value)`
       - `neighborhood` filter: `ilike(voters.neighborhood, '%value%')` or similar
       - `tags` filter: use `sql` helper for array contains (Postgres `@>` operator or `&&`)
       - `optInStatus` filter: `eq(voters.optInStatus, value)`
       - `engagementScore` filter: `gte(voters.engagementScore, min)` and/or `lte(..., max)`
     - AND/OR logic: if filters have `operator: 'AND'`, combine with `and()`. If `operator: 'OR'`, combine with `or()`
     - Return count and matching voter IDs

  3. **On segment save (POST without ?action=preview):**
     - After creating the segment record, call `setSegmentVoters(segmentId, voterIds)` from db-segments.ts
     - Call `updateSegmentCount(segmentId, voterIds.length)` to persist the count
     - This materializes the audience into the junction table so campaigns can resolve it

  **Page: Real audience count + dynamic filters**
  1. In `segmentacao/page.tsx`, replace the mock/random audience count:
     - When user changes filters, debounce (500ms) then call the preview endpoint
     - Display returned count as the audience size
     - Show "Calculando..." while the request is in flight
  2. **Coverage percentage:** Replace `audienceCount / 1000` with `audienceCount / totalVoterCount` where totalVoterCount comes from `GET /api/voters` total (or a separate count endpoint)
  3. **Dynamic filter options:** On page load, fetch real values:
     - `GET /api/voters` already returns voters — extract unique zones, sections, neighborhoods, tags from the data
     - OR better: add a `GET /api/segments?action=filter-options` endpoint that returns `{ zones: string[], neighborhoods: string[], tags: string[] }` by running `SELECT DISTINCT zone FROM voters`, etc.
     - Populate the filter dropdowns with these real values instead of hardcoded arrays
  </action>
