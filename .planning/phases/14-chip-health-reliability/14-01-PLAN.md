**Schema changes (src/db/schema.ts):**

1. Replace the chip status enum. The current enum is `chipStatusEnum = pgEnum('chip_status', ['connected', 'disconnected', 'warming'])`. Change to:
```
chipStatusEnum = pgEnum('chip_status', ['healthy', 'degraded', 'cooldown', 'quarantined', 'banned', 'warming_up', 'disconnected'])
```

NOTE: Since we're on Supabase/Drizzle, we need to handle the migration carefully. If direct ALTER TYPE is problematic, we can add a separate `chipHealthStatus` text field and keep the old enum. **Prefer** adding a new text field `healthStatus` with a default of 'disconnected' to avoid migration issues with the existing enum. Keep the old `status` field for backward compatibility but mark it deprecated.

2. Add new fields to chips table:
```typescript
healthStatus: text('health_status').default('disconnected').notNull(), // 7-state health
messagesSentToday: integer('messages_sent_today').default(0).notNull(),
messagesSentThisHour: integer('messages_sent_this_hour').default(0).notNull(),
dailyLimit: integer('daily_limit').default(200).notNull(),
hourlyLimit: integer('hourly_limit').default(25).notNull(),
lastHealthCheck: timestamp('last_health_check', { withTimezone: true }),
lastWebhookEvent: timestamp('last_webhook_event', { withTimezone: true }),
cooldownUntil: timestamp('cooldown_until', { withTimezone: true }),
bannedAt: timestamp('banned_at', { withTimezone: true }),
errorCount: integer('error_count').default(0).notNull(),
blockRate: integer('block_rate').default(0), // percentage * 100 (e.g., 250 = 2.5%)
assignedSegments: text('assigned_segments').array(), // segment IDs this chip handles
```

**db-chips.ts updates:**
- Add `updateChipHealth(chipId, healthFields)` function for partial updates on health fields
- Add `getHealthyChips()` function that returns chips with healthStatus in ('healthy', 'degraded') and messagesSentToday < dailyLimit
- Add `incrementChipCounter(chipId, field: 'messagesSentToday' | 'messagesSentThisHour')` using SQL increment
- Add `resetDailyCounters()` to zero out all chips' messagesSentToday and messagesSentThisHour

**Create src/app/api/cron/chip-health/route.ts:**

A GET endpoint (cron-triggered) that:
1. Auth check: Requires `CRON_SECRET` header (same pattern as existing crons)
2. Loads all enabled chips from DB
3. Loads config (evolutionApiUrl, evolutionApiKey) 
4. For each chip with an instanceName:
   a. Call `getConnectionState()` from evolution.ts
   b. Check lastWebhookEvent staleness (>2min = degraded)
   c. Determine health:
      - connectionState='open' AND webhook fresh → 'healthy'
      - connectionState='open' AND webhook stale → 'degraded' 
      - connectionState='connecting' → attempt restartInstance(), wait 5s, recheck
      - connectionState='close' → attempt restartInstance(), wait 5s, recheck. If still close → 'disconnected'
   d. If chip was 'disconnected' and restart fails 3 times (errorCount >= 3) → mark as 'quarantined'
   e. Check if cooldownUntil has passed → restore to 'healthy'
   f. Update chip in DB with new healthStatus + lastHealthCheck timestamp
5. Return JSON summary: { checked: N, healthy: N, degraded: N, disconnected: N, quarantined: N }

**Webhook hardening (src/app/api/webhook/route.ts):**

Keep ALL existing logic but add:
1. On CONNECTION_UPDATE: Extract `statusReason` from data (200, 401, 408, 515). 
   - statusReason 401 (logged out) → set healthStatus='disconnected', log warning
   - statusReason 408 (timeout) → attempt restart via cron (don't block webhook)
   - statusReason 515 (restart needed) → same as 408
   - Update `lastWebhookEvent` timestamp on EVERY webhook event received
2. Add handler for MESSAGES_UPDATE event (placeholder for Phase 17 — just log the event and status code for now, store nothing yet). This ensures the webhook is ready to receive delivery status updates.
3. Add handler for GROUP_PARTICIPANTS_UPDATE event (placeholder for Phase 16 — just log the event for now).
4. Resolve chip from instanceName and set `lastWebhookEvent = new Date()` on EVERY event type (not just connection.update).

**Generate migration SQL:**
Create `drizzle/0002_chip_health_fields.sql` with ALTER TABLE statements to add the new columns. Use `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` for idempotency.
