<objective>
Establish the backend foundation for Phase 10 so chat surfaces can subscribe to persisted conversation updates over SSE without changing the canonical write paths.

Purpose: the current system already stores inbound webhook events, conversation status changes, and outbound agent replies in Postgres. This plan exposes those stored changes through one authenticated stream contract with cursor resume and heartbeat handling.

Output: `/api/conversations/stream` exists, authenticates with the existing session cookie, replays deltas since a cursor, and emits queue/thread events derived from database state.
</objective>

  Create the production-facing SSE endpoint that chat clients will consume:

  1. Add `GET /api/conversations/stream` as a Route Handler returning `text/event-stream`.
  2. Authenticate the request with the same cookie/session pattern already used by the conversations APIs and return `401` for unauthenticated callers.
  3. Parse supported query params:
     - `status`
     - `conversationId`
     - `voterId`
     - `since`
  4. On connect, emit either a lightweight `connected`/`snapshot-ready` event or a first delta replay based on the supplied cursor.
  5. While the request is open, run a bounded server-side loop that:
     - checks for new conversation/message deltas every 1-2 seconds
     - emits only the normalized SSE events produced by the shared helper
     - sends heartbeat frames often enough to keep the connection alive behind intermediaries
     - stops immediately when `request.signal.aborted` fires
  6. Keep this plan scoped to SSE delivery only. Do not replace existing REST mutation routes or introduce provider-driven sockets.
  </action>
