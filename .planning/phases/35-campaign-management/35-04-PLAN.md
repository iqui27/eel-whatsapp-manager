  <name>Task 1: Refactor send-queue to read per-campaign config</name>
  <files>
    src/app/api/cron/send-queue/route.ts
    src/lib/db-campaigns.ts
  </files>
  <action>
    **Major refactor of `send-queue/route.ts`:**

    1. **Remove hardcoded constants** at the top (BATCH_SIZE, MIN_DELAY_MS, MAX_DELAY_MS, TIME_WINDOW_START, TIME_WINDOW_END, TYPING_DELAY_MIN, TYPING_DELAY_MAX). Replace with per-campaign config.

    2. **Group queued messages by campaignId** before processing. For each campaign batch:
       a. Load campaign config: `getCampaign(campaignId)` — this returns the full campaign row with all new send config fields.
       b. Use campaign's `batchSize` (or default 10 if null) to limit messages per campaign per run.
       c. Use campaign's `windowStart`/`windowEnd` to check time window (parse "HH:MM" string to hour comparison).
       d. Use campaign's `minDelayMs`/`maxDelayMs` for the inter-message delay.

    3. **Presence simulation before each message:**
       - If Evolution API supports presence simulation, call it. The Evolution v2 API has no direct "typing" endpoint for individual chats, so simulate via delay:
       ```ts
       const typingDelay = randomInt(campaign.typingDelayMin ?? 2000, campaign.typingDelayMax ?? 5000);
       await sleep(typingDelay);
       ```
       This happens BEFORE the `sendText()` call — simulating the time a human would take to type.

    4. **Rest pauses:**
       - Track messages sent in this run per campaign with a counter.
       - Every `campaign.restPauseEvery` messages (default 20), sleep for `campaign.restPauseDurationMs` (default 3 min).
       - Every `campaign.longBreakEvery` messages (default 100), sleep for `campaign.longBreakDurationMs` (default 15 min).
       - Note: These are per-cron-run counters, not persistent. Since cron runs are short (~1-5min), the rest pauses within a single run create natural breaks.

    5. **Circuit breaker:**
       - Track failures per campaign in this run.
       - If `failedCount / totalAttempted * 100 >= campaign.circuitBreakerThreshold` (default 5%) AND totalAttempted >= 5 (minimum sample), pause the campaign:
         ```ts
         await updateCampaign(campaignId, { status: 'paused' });
         syslog({ level: 'warn', category: 'campaign', message: `Campaign ${campaignId} circuit-breaker tripped`, details: { failedCount, totalAttempted, threshold: campaign.circuitBreakerThreshold } });
         ```
       - Skip remaining messages for that campaign in this run.

    6. **Backward compatibility:** For campaigns without the new fields (all fields have DB defaults), behavior should match current hardcoded values exactly.

    **db-campaigns.ts:** Add a `getCampaign(id)` function if not already present (or reuse existing). It should return the full campaign row including new send config fields.
  </action>

<task type="auto">
  <name>Task 2: Update chip router to respect campaign chip selection</name>
  <files>src/lib/chip-router.ts</files>
  <action>
    Update `selectBestChip()` to accept an optional `campaignChipConstraints` parameter:

    ```ts
    interface CampaignChipConstraints {
      selectedChipIds?: string[];    // Only consider these chips
      chipStrategy?: 'round_robin' | 'least_loaded' | 'affinity';
      maxDailyPerChip?: number;      // Campaign-level daily limit (may differ from chip's own limit)
      maxHourlyPerChip?: number;     // Campaign-level hourly limit
      pauseOnChipDegraded?: boolean; // Skip degraded chips if true
    }
    ```

    Logic changes:
    1. **Filter by selectedChipIds** — If `selectedChipIds` is provided and non-empty, only consider those chips. Otherwise, consider all healthy chips (current behavior).

    2. **Respect pauseOnChipDegraded** — If true (default), exclude chips with healthStatus='degraded' in addition to the current exclusions (quarantined, banned, disconnected).

    3. **Apply campaign-level limits** — Before selecting a chip, check if it has exceeded `maxDailyPerChip` or `maxHourlyPerChip` from the campaign constraints. These may be lower than the chip's own global limits.

    4. **Strategy selection:**
       - `round_robin` (default): Distribute messages evenly across eligible chips. Track index in a module-level variable (reset each cron run).
       - `least_loaded`: Sort by (dailyLimit - messagesSentToday) descending, pick the one with most remaining capacity.
       - `affinity`: Current behavior — use the existing scoring algorithm which considers segment affinity.

    Keep the existing `selectBestChip(segmentId)` signature as a backward-compatible wrapper:
    ```ts
    export function selectBestChip(segmentId?: string | null, constraints?: CampaignChipConstraints) { ... }
    ```
  </action>
