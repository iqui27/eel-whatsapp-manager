<task type="auto">
  <name>Task 1: Add campaign chip stats API endpoint</name>
  <files>src/app/api/campaigns/route.ts</files>
  <action>
    When GET `/api/campaigns?id={campaignId}` returns a single campaign, enrich the response with chip-level anti-ban stats:

    1. Read the campaign's `selectedChipIds` array. If empty or null, fall back to ALL enabled chips.

    2. For each selected chip, fetch from the chips table:
       - `name`, `instanceName`, `healthStatus`
       - `messagesSentToday`, `messagesSentThisHour`, `dailyLimit`, `hourlyLimit`
       - `errorCount`
       - `proxyHost` (just presence — show "Proxy: ✓" or "Sem proxy")

    3. For each chip, compute from the message_queue table:
       - `errorRate`: Count messages with status='failed' for this chip+campaign / total attempted for this chip+campaign. Use a single SQL query grouped by chipId.
       - `sentToday`: Count messages sent today for this campaign via this chip.

    4. Derive warm-up stage per chip based on chip creation date or first message date:
       - Day 1-3: "Fase 1 — Aquecimento Inicial" (max 20/day)
       - Day 4-7: "Fase 2 — Acelerando" (max 50/day)
       - Week 2-4: "Fase 3 — Ativo" (max 200/day)
       - Month 2+: "Fase 4 — Maduro" (max 350/day)
       - Month 3+: "Fase 5 — Veterano" (max 500/day)
       Use the chip's `createdAt` field. If `createdAt` doesn't exist, use a fallback heuristic (assume mature).

    5. Add a `chipStats` array to the campaign response:
       ```ts
       chipStats: Array<{
         chipId: string;
         chipName: string;
         healthStatus: string;
         warmupStage: string;
         recommendedDailyMax: number;
         messagesSentToday: number;
         messagesSentThisHour: number;
         dailyLimit: number;
         hourlyLimit: number;
         errorRate: number;        // 0-100 percentage
         hasProxy: boolean;
         sentForCampaign: number;  // Total sent for THIS campaign via this chip
       }>
       ```

    6. Add a `circuitBreakerStatus` field:
       ```ts
       circuitBreakerStatus: 'active' | 'tripped' | 'disabled'
       ```
       - `disabled`: circuitBreakerThreshold is 0 or null
       - `tripped`: campaign status is 'paused' (tripped by circuit breaker logic in 35-04)
       - `active`: threshold is set and campaign is not paused

    Keep this enrichment ONLY for single-campaign GET (when `id` param is provided), NOT for list queries.
  </action>

  </verify>
