<objective>
Add missing database indexes and convert voter queries from unbounded SELECT * to SQL-level pagination with LIMIT/OFFSET.

Purpose: `loadVoters()` and `filterVoters()` currently return the entire voters table (could be 100K+ rows), then the API paginates in JavaScript. This wastes DB bandwidth, memory, and CPU. SQL-level pagination with proper indexes makes queries O(page_size) instead of O(total_rows).
Output: Missing indexes added, voter queries properly paginated at DB level, Drizzle migration applied.
</objective>

1. **`src/lib/db-voters.ts`** — Add a new `filterVotersPaginated()` function that returns `{ data: Voter[], total: number }` with SQL LIMIT/OFFSET:

```ts
export interface PaginatedResult<T> {
  data: T[];
  total: number;
}

export async function filterVotersPaginated(
  filters: VoterFilters,
  pagination: { limit: number; offset: number },
): Promise<PaginatedResult<Voter>> {
  const conditions = [];

  // ... same filter conditions as filterVoters() ...
  // (copy the existing condition-building logic from filterVoters)

  // For segment filter, build the subquery differently to work with count
  let segmentVoterIds: string[] | null = null;
  if (filters.segmentId) {
    const voterIdsInSegment = await db
      .select({ voterId: segmentVoters.voterId })
      .from(segmentVoters)
      .where(eq(segmentVoters.segmentId, filters.segmentId));
    segmentVoterIds = voterIdsInSegment.map((r) => r.voterId);
    if (segmentVoterIds.length === 0) return { data: [], total: 0 };
    conditions.push(inArray(voters.id, segmentVoterIds));
  }

  const where = conditions.length > 0 ? and(...conditions) : undefined;

  // Get count (single integer, no data transfer)
  const countResult = await db
    .select({ count: sql<number>`count(*)::int` })
    .from(voters)
    .where(where);
  const total = countResult[0]?.count ?? 0;

  if (total === 0) return { data: [], total: 0 };

  // Get paginated data
  const data = await db
    .select()
    .from(voters)
    .where(where)
    .orderBy(desc(voters.createdAt))
    .limit(pagination.limit)
    .offset(pagination.offset);

  return { data, total };
}
```

Also modify `loadVoters()` to accept an optional limit parameter:
```ts
export async function loadVoters(limit?: number): Promise<Voter[]> {
  const query = db.select().from(voters).orderBy(desc(voters.createdAt));
  if (limit) return query.limit(limit);
  return query;
}
```

**Keep the existing `filterVoters()` function unchanged** — it's still called by callers that need all results (compliance, conversations). Add the new paginated version alongside it.

2. **`src/app/api/voters/route.ts`** — Replace the current pattern:
```ts
// BEFORE (loads ALL voters then slices in JS):
const allData = await filterVoters({ ... });
const scopedData = allData.filter(v => isVoterInScope(auth.actor, v));
const data = scopedData.slice(offset, offset + limit);
```

With SQL-level pagination:
```ts
// AFTER (SQL pagination):
import { filterVotersPaginated } from '@/lib/db-voters';

const { data: paginatedData, total } = await filterVotersPaginated(
  { search: query ?? undefined, tag: ..., ... },
  { limit, offset },
);
// Apply scope filter on paginated results (small set)
const scopedData = paginatedData.filter(v => isVoterInScope(auth.actor, v));
```

**Note on scope filtering:** The `isVoterInScope` check happens after SQL pagination. For the common case (admin user with no region scope), this is a no-op pass-through. For scoped users, the total count may be slightly off, but this is acceptable — exact count with scope would require a complex SQL condition that matches the runtime scope logic. The current approach is correct for MVP with single-user deployment.
