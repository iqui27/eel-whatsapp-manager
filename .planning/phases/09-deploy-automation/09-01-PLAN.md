<task type="auto">
  <name>Task 1: Create server-side deploy and rollback scripts</name>
  <files>scripts/deploy.sh, scripts/rollback.sh</files>
  <action>
Create `scripts/deploy.sh` — the server-side deploy script that GitHub Actions will execute via SSH.

The script MUST:
1. Set strict mode (`set -euo pipefail`) and log every step with timestamps
2. Define constants: `APP_DIR=/opt/zap-app`, `PM2_APP=zap-eel`, `NODE_ENV=production`
3. Navigate to `$APP_DIR`
4. **Backup current build** before anything: `cp -r .next/standalone .next/standalone.bak 2>/dev/null || true`
5. `git fetch origin main && git reset --hard origin/main` (always match remote exactly)
6. `npm ci --production=false` (need devDeps for build)
7. `npm run build` — if this FAILS, the script must exit WITHOUT restarting PM2 (zero-downtime safety)
8. Only after successful build: `pm2 restart $PM2_APP --update-env`
9. Wait 5 seconds, then `pm2 status $PM2_APP` to confirm it's running
10. `curl -sf http://localhost:3002 > /dev/null` — health check, warn if fails but don't rollback automatically
11. Print summary: git SHA, build time, status

Create `scripts/rollback.sh` — instant rollback:
1. Restores `.next/standalone.bak` → `.next/standalone`
2. `pm2 restart $PM2_APP`
3. Confirms health check

Both scripts must be executable (`chmod +x`).

Important: Do NOT use `git pull` (can fail with diverged history). Use `git fetch + reset --hard` for reliable deploys.
  </action>

- Steps:
  1. **Checkout** — `actions/checkout@v4` (only needed for workflow file itself)
  2. **Deploy via SSH** — Use `appleboy/ssh-action@v1.2.0` with:

     - `key: ${{ secrets.CONTABO_SSH_KEY }}` (private key content)
     - `port: 22`
     - `command_timeout: 10m` (build can take a while)
     - `script:` runs `/opt/zap-app/scripts/deploy.sh`
  3. **Notify on failure** — Add `if: failure()` step that creates a GitHub deployment status or just echoes failure

**Required GitHub Secrets** (user must configure):
- `CONTABO_HOST` — `193.187.129.114`
- `CONTABO_USER` — `root`
- `CONTABO_SSH_KEY` — private SSH key content (from Bitwarden "Contabo Pessoal")

Add a comment block at the top of the workflow explaining:
- What secrets are needed
- How to add them (Settings → Secrets → Actions)
- What the workflow does step by step

Do NOT use `docker` or `rsync` approaches. The server already has the git repo cloned. SSH + script execution is the simplest and most reliable approach.
